## 1. 模块概述

### 定位

工具召回排序模块位于**意图分类器**和 **MCP Subagent 执行层**之间。当意图分类器将用户 query 判定为 `tool_call` 类型后，该模块负责从 50 个工具中精准召回用户所需的工具，交由 LLM 进行最终 Tool Call 决策。

### 核心目标

| 目标 | 指标 |
|------|------|
| 精准召回用户意图对应的工具 | Recall@5 ≥ 89% |
| 将正确工具排在靠前位置 | MRR ≥ 0.83 |
| 端到端延迟可控 | 50 个工具检索 < 500ms |

### 对比基线

| 方法 | Recall@5 | MRR | 说明 |
|------|----------|-----|------|
| BM25（jieba 分词） | ~61% | ~0.52 | 传统关键词匹配 baseline |
| BGE-M3 向量检索 | ~81% | ~0.74 | 语义向量粗排 |
| **BGE-M3 + bge-reranker** | **≥89%** | **≥0.83** | **本模块方案** |

## 2. 系统架构

### 数据流

```
用户 query: "帮我设置明天早上8点的闹钟"
        │
        ▼
┌───────────────────────────────┐
│   ToolEmbedder (BGE-M3)       │
│   query → 1024 维归一化向量     │
└──────────────┬────────────────┘
               │
               ▼
┌───────────────────────────────┐
│   FaissToolStore (IndexFlatIP) │
│   向量检索 → Top-20 候选工具    │
│   余弦相似度粗排                │
└──────────────┬────────────────┘
               │
               ▼
┌───────────────────────────────┐
│   ToolReranker                │
│   (bge-reranker-v2-m3)        │
│   交叉编码精排 → Top-5         │
└──────────────┬────────────────┘
               │
               ▼
┌───────────────────────────────┐
│   返回 Top-5 工具列表          │
│   → MCPExecutor.run()         │
└───────────────────────────────┘
```

### 两阶段检索策略

| 阶段 | 模型 | 方法 | 输入 | 输出 |
|------|------|------|------|------|
| 粗排 | BGE-M3 (bi-encoder) | FAISS 内积检索 | query 向量 vs 50 个工具向量 | Top-20 候选 |
| 精排 | bge-reranker-v2-m3 (cross-encoder) | 逐对打分 | (query, 工具文本) × 20 对 | Top-5 结果 |

**为什么需要两阶段？**
- Bi-encoder（BGE-M3）速度快但精度有限，适合从大量候选中粗筛
- Cross-encoder（reranker）精度高但速度慢，适合对少量候选精排
- 两阶段结合兼顾速度与精度

## 4. 模块详细设计

### 4.1 BGE-M3 编码器 (`embedder.py`)

**职责：** 将文本编码为 1024 维归一化向量。

**模型：** `BAAI/bge-m3`（~2.2GB），通过 `FlagEmbedding.BGEM3FlagModel` 加载。

**设计要点：**
- 单例模式，首次调用时懒加载，后续复用同一实例
- BGE-M3 默认返回 L2 归一化向量，可直接用于 FAISS 内积检索（余弦相似度）
- 设置 `HF_ENDPOINT=https://hf-mirror.com` 使用镜像源下载模型

**接口：**

```python
class ToolEmbedder:
    @classmethod
    def get_instance(cls) -> "ToolEmbedder"

    def encode_query(self, query: str) -> np.ndarray:
        """编码单条 query → (1024,) 向量"""

    def encode_documents(self, texts: list[str]) -> np.ndarray:
        """批量编码 → (N, 1024) 向量矩阵"""
```

### 4.2 FAISS 向量库 (`faiss_store.py`)

**职责：** 加载预构建索引，执行向量近邻搜索。

**索引类型：** `faiss.IndexFlatIP`（内积，归一化向量 = 余弦相似度）

**为什么用 IndexFlatIP？**
- 50 个向量，暴力搜索耗时微秒级
- 无需 IVF（倒排文件）或 HNSW（近似最近邻），索引构建和查询都更简单
- 精确搜索，无召回损失

**接口：**

```python
class FaissToolStore:
    def load(self, index_dir: Path, registry_path: Path):
        """从磁盘加载索引和工具元数据"""

    def search(self, query_vec: np.ndarray, top_k: int = 20) -> list[dict]:
        """返回 [{tool_id, score, tool_meta}]"""

    def get_tool_text(self, tool_id: str) -> str:
        """获取工具复合文本（供 reranker 使用）"""
```

### 4.3 BGE Reranker (`reranker.py`)

**职责：** 对粗排候选进行交叉编码精排。

**模型：** `BAAI/bge-reranker-v2-m3`（~1.1GB），通过 `FlagEmbedding.FlagReranker` 加载。

**工作原理：**
- Cross-encoder 将 (query, tool_text) 拼接后一起编码，计算相关性分数
- 比 bi-encoder 精度更高，但计算量大（所以只对 Top-20 候选精排）

**接口：**

```python
class ToolReranker:
    @classmethod
    def get_instance(cls) -> "ToolReranker"

    def rerank(self, query: str, candidates: list[dict], top_k: int = 5) -> list[dict]:
        """精排并返回 top-k 结果，按 rerank_score 降序"""
```

### 4.4 异步编排层 (`pipeline.py`)

**职责：** 将三个组件串联为异步管线，供 FastAPI 路由调用。

**关键设计：**
- 所有 CPU/GPU 密集操作通过 `asyncio.to_thread()` 包裹，不阻塞事件循环
- 启动时调用 `initialize()` 预加载模型，避免首次请求延迟
- 对外只暴露 `retrieve(query, top_k)` 一个接口

**接口：**

```python
class RetrievalPipeline:
    @classmethod
    async def initialize(cls):
        """启动时预加载所有模型和索引"""

    @classmethod
    async def retrieve(cls, query: str, top_k: int = 5) -> list[dict]:
        """异步检索 top-k 工具，返回 registry.json 格式的工具列表"""
```

**调用示例：**

```python
# FastAPI 启动时
@app.on_event("startup")
async def startup():
    await RetrievalPipeline.initialize()

# 路由中使用
tools = await RetrievalPipeline.retrieve("帮我查明天北京的天气", top_k=5)
```

## 5. 工具文本表示策略

每个工具在向量化前会构建一段**复合文本**，格式为：

```
{name}。{description}。示例：{example1}; {example2}; {example3}
```

**示例：**

```
查询天气。查询指定城市在指定日期的天气情况，包括温度、湿度、风速、天气状况。示例：明天北京天气怎么样; 上海今天下雨吗; 查一下深圳周末天气
```

**为什么包含 examples？**
- 用户 query 是口语化的（"帮我查查明天天气"），而 description 是正式的（"查询指定城市在指定日期的天气情况"）
- examples 桥接了两者的词汇鸿沟，显著提升语义匹配效果
- `。` 和 `示例：` 作为分隔符，帮助 BGE-M3 tokenizer 识别段落边界

## 6. 离线索引构建

**脚本：** `scripts/build_index.py`

**运行方式：**

```bash
python scripts/build_index.py
```

**执行流程：**

1. 读取 `tools/registry.json`（50 个工具）
2. 为每个工具构建复合文本
3. 用 BGE-M3 批量编码为 1024 维向量
4. 构建 FAISS IndexFlatIP 索引
5. 保存到 `data/tool_index/`：
   - `tools.index` — FAISS 二进制索引
   - `tool_ids.json` — 工具 ID 有序列表（映射 FAISS 内部索引）
   - `tool_texts.json` — 工具 ID → 复合文本（供 reranker 和调试使用）

**特点：** 幂等执行，可重复运行，输出会被覆盖。

## 7. 评测方案

**脚本：** `eval/eval_retrieval.py`

**测试集：** `data/tool_test/retrieval_testset.json`（50 条，每条含 1 个正例 + 2 个难负例）

**评测模式：**

```bash
python eval/eval_retrieval.py                # 全量对比（BM25 + FAISS + Reranker）
python eval/eval_retrieval.py --mode bm25    # 仅 BM25 baseline
python eval/eval_retrieval.py --mode faiss   # 仅 FAISS 向量检索
python eval/eval_retrieval.py --mode full    # FAISS + Reranker 完整管线
```

**输出示例：**

```
============================================================
  BM25 Baseline
============================================================
  Recall@1     0.5200
  Recall@3     0.6800
  Recall@5     0.6100
  MRR          0.5200

============================================================
  FAISS (BGE-M3)
============================================================
  Recall@1     0.7400
  Recall@3     0.8600
  Recall@5     0.8100
```

**评测指标：**

| 指标 | 公式 | 含义 |

## 10. 面试考点

**Q: 为什么用两阶段检索而不是直接用 reranker？**
> Reranker 是 cross-encoder，需要对每个 (query, tool) 对做前向推理，50 个工具就是 50 次推理，延迟不可接受。Bi-encoder 可以一次编码所有工具向量，FAISS 检索只需微秒级。两阶段粗排+精排兼顾速度与精度。

**Q: 为什么用 IndexFlatIP 而不是 IVF 或 HNSW？**
> 只有 50 个向量，暴力搜索耗时微秒级。IVF 需要聚类训练，HNSW 需要构建图结构，都是大材小用。IndexFlatIP 简单、精确、无召回损失。

**Q: BGE-M3 和 bge-reranker 有什么区别？**
> BGE-M3 是 bi-encoder，query 和 document 独立编码，通过向量内积计算相似度，速度快。bge-reranker 是 cross-encoder，将 query 和 document 拼接后一起编码，能捕捉更细粒度的交互特征，精度更高但速度慢。

**Q: 工具文本为什么包含 examples？**
> 用户 query 是口语化的，工具 description 是正式的，两者存在词汇鸿沟。examples 用自然语言桥接了这个鸿沟，让 bi-encoder 能更好地理解用户意图。
